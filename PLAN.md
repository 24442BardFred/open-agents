Summary: Rebuild the terminal architecture around a persistent in-sandbox terminal gateway that supports both browser and native transports, instead of the current raw WebSocket-to-PTY bridge. The recommended end state is a terminal multiplexer/gateway process in the sandbox with a browser client for the UI today and a first-class native Mosh/SSH path later, because that is the only path that can realistically deliver excellent TUI support and much better perceived latency.

Context: The current implementation in [apps/web/lib/sandbox/terminal/server-script.ts](#workspace-file=apps/web/lib/sandbox/terminal/server-script.ts) is a direct remote-echo design: the browser sends keystrokes immediately over WebSocket (`term.onData(... socket.send(data))`), the server writes them directly to the PTY (`ptyProcess.write(message)`), and PTY output is sent back and rendered (`ws.send(data)` -> `term.write(event.data)`). That architecture guarantees typing feel is bounded by RTT. [apps/web/lib/sandbox/terminal/bootstrap.ts](#workspace-file=apps/web/lib/sandbox/terminal/bootstrap.ts) bootstraps a custom Node terminal server inside the sandbox, installs `@lydell/node-pty`, `ghostty-web`, and `ws`, and exposes it over sandbox port routing. Ghostty Web’s docs describe a browser VT implementation with xterm-compatible APIs, not a latency-hiding transport. Mosh’s docs explicitly attribute its feel to speculative local echo and predictive state reconciliation, including support for full-screen apps. The latch project demonstrates the architectural shape we actually want: a persistent terminal gateway/multiplexer process with multiple transports (web, SSH, Mosh) sharing the same session model.

Approach: Throw away the current custom one-off terminal server architecture and replace it with a dedicated terminal gateway running inside the sandbox. That gateway becomes the source of truth for terminal sessions and exposes: (1) a browser transport for the Open Harness UI, and (2) a future native transport path for external Mosh/SSH clients. For the browser path, do not try to fake Mosh over the existing raw PTY bridge. Instead, build the browser transport against a gateway that owns session state, screen state, resize handling, reconnect behavior, auth, and future predictive hooks. The UI should embed the terminal directly, but the real architectural shift is moving from “web page controlling a shell directly” to “terminal gateway controlling long-lived sessions and transports.” If we want best-in-class typing feel, phase 2 should either adopt an existing gateway/multiplexer with Mosh support or implement prediction/state-sync semantics at the gateway layer rather than inside the React app.

Changes:
- [apps/web/lib/sandbox/terminal/bootstrap.ts](#workspace-file=apps/web/lib/sandbox/terminal/bootstrap.ts) - replace bootstrap logic so it provisions and manages a persistent terminal gateway binary/service rather than writing a custom ad hoc Node server script and npm-installing browser assets inside `/tmp`.
- [apps/web/lib/sandbox/terminal/server-script.ts](#workspace-file=apps/web/lib/sandbox/terminal/server-script.ts) - remove this file entirely in favor of a real gateway runtime with a clearer protocol boundary.
- [packages/sandbox/interface.ts](#workspace-file=packages/sandbox/interface.ts) - extend sandbox capabilities as needed for terminal gateway lifecycle, long-lived processes, and potentially richer transport metadata.
- [packages/sandbox/vercel/sandbox.ts](#workspace-file=packages/sandbox/vercel/sandbox.ts) - add any needed helpers for stable long-lived detached terminal services and exposed transport endpoints.
- [apps/web/app/api/sessions/[sessionId]/terminal/route.ts](#workspace-file=apps/web/app/api/sessions/[sessionId]/terminal/route.ts) - change launch semantics from “start a shell page” to “issue a browser session token/endpoint for a terminal gateway session.”
- [apps/web/app/sessions/[sessionId]/chats/[chatId]/terminal-panel.tsx](#workspace-file=apps/web/app/sessions/[sessionId]/chats/[chatId]/terminal-panel.tsx) - simplify into a browser client for the gateway protocol rather than owning terminal transport behavior itself.
- New terminal gateway area under `apps/web/lib/sandbox/terminal/` and/or a new package - define session model, auth token issuance, browser transport client, reconnect semantics, and future native transport support.
- Possibly a new sandbox-resident binary/runtime bundle checked into the repo or built as part of the app - preferred over in-sandbox `npm install` at runtime.

Verification:
- End-to-end: open a terminal in the app, disconnect/reconnect, and confirm the session persists rather than creating a fresh PTY every time.
- TUI validation: run `vim`, `nvim`, `less`, and shell line editing under induced latency and confirm behavior remains correct.
- Transport validation: verify multiple attachments to the same terminal session if the gateway supports it.
- Lifecycle validation: restart/reconnect sandbox and confirm gateway bootstrap is deterministic and fast.
- Commands after implementation: `bun run ci`, plus manual terminal interaction tests against the running app.
- Stretch verification for the best architecture: test an external native transport path (Mosh or SSH) attaching to the same sandbox session model, if included in the phase.
